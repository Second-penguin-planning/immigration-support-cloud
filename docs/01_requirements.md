# 要件定義

## 1. 機能要件

### 1.1 ログイン機能（Phase3）

- メールアドレス + パスワードによるログイン（メール認証必須。未認証アカウントはログイン不可）
- パスワードリセット（メールにリセットリンクを送付、有効期限付きトークン）
- 二段階認証（TOTP）に対応できる設計とする（MVPでは任意有効化、将来必須化も想定）
- 権限管理（3ロール）
  - **管理者 (admin)**: 全機能利用可、ユーザー招待・権限変更、監査ログ閲覧
  - **スタッフ (staff)**: 顧客管理・申請書類作成等の実務全般（ユーザー管理は不可）
  - **閲覧のみ (viewer)**: 参照のみ。作成・更新・削除・ダウンロード操作は不可
- ログイン試行のレート制限、監査ログへの記録（ログイン成功/失敗、パスワード変更等）

### 1.2 顧客管理（Phase4）

- 法人情報（法人名、所在地、担当者、連絡先、契約状況等）の登録・編集・削除
- 外国人情報（氏名、国籍、生年月日、旅券番号、在留カード番号等）の登録・編集・削除
- 在留資格（資格の種類、許可年月日、在留期限）の管理。1名の外国人に対して在留資格の履歴を複数保持できる
- 期限（在留期限・各種届出期限）の一覧・検索
- 担当者（社内の誰がその顧客を担当しているか）の割り当て
- 検索（法人名・氏名・在留カード番号・期限範囲・担当者などの複合条件）
- CSVダウンロード（検索結果を任意の列構成でCSV出力）

### 1.3 Excelアップロード（Phase4〜Phase6と連携）

- ユーザーが既存Excel（顧客一覧・外国人名簿等）をアップロード
- シート構造を解析し、必要項目（氏名・国籍・在留資格・期限等）を自動抽出
- 抽出結果をプレビュー表示し、確認後にDBへ保存
- 入力エラー（必須項目欠落・日付形式不正・在留資格コード不正等）を行単位で表示し、修正を促す
- 取り込みテンプレート（列定義）は複数パターンに対応できる構造とする

### 1.4 CSV生成（Phase5）

- 入管オンライン提出用CSVをボタン一つで生成・ダウンロード
- 出力項目・列順・エンコーディング（Shift_JIS等）はテンプレートとして定義し、
  入管側フォーマット変更時にテンプレートの追加・変更のみで対応できる構造とする
- 生成前にバリデーション（必須項目・文字種・桁数）を行い、不備があれば生成をブロックしエラー箇所を提示する

### 1.5 PDF管理（Phase5〜）

- 添付書類（在留カード写し、雇用契約書等）のPDF/画像アップロード
- アップロード時にファイル名を規則的に自動変更（例: `{外国人氏名}_{書類種別}_{年月日}.pdf`）
- 書類種別ごとに必須/任意を定義し、不足書類を一覧表示
- 顧客・外国人単位での書類一覧表示、ダウンロード

### 1.5b 在留資格認定証明書交付申請（Phase9）

- 初めて在留資格を取得する外国人向けに、交付申請の準備〜結果までを管理する
  （在留カード番号がまだ存在しない段階から利用できる、既存の「在留資格」管理とは別の機能）
- ステータス管理: 準備中 → 提出済み → 交付 / 不交付 / 取下げ
- 申請予定日・提出日・結果通知日を記録
- 交付された場合、在留カード番号・許可年月日・在留期限を入力して既存の「在留資格の履歴」へ変換・確定する

### 1.6 定期届出（Phase7）

- 特定技能定期届出（四半期ごと等）について、前回提出データをコピーして新しいドラフトを作成
- 変更点のみ入力する差分入力UI（変更なしの項目は前回値をそのまま保持）
- 面談記録（実施日、面談者、内容メモ）の登録
- 支援実施状況（生活オリエンテーション、日本語学習支援等の実施有無・実施日）の登録
- 上記をもとに届出ドラフト（入管提出様式に対応する構造化データ）を作成

### 1.7 AI補助（Phase6）

- OCR: アップロードされた書類画像/PDFから文字情報を抽出
- PDF読取: 既存PDF（雇用契約書等）からテキストレイヤーを抽出し構造化
- 不足項目抽出: OCR/PDF読取結果と必須項目を突き合わせ、不足項目を提示
- 誤入力検知: 日付の前後関係、在留資格と職種の整合性等のルールベース検証 + AIによる違和感検知
- 入力候補表示: 過去データや抽出結果から入力候補をサジェスト

### 1.8 ダッシュボード（Phase4〜Phase8で段階拡張）

- 期限管理: 期限までの残日数で「30日」「14日」「7日」「今日」に分類表示
- 通知表示: 上記期限に基づくアプリ内通知（メール通知等は将来拡張）

## 2. 非機能要件

| 区分 | 要件 |
| --- | --- |
| 型安全性 | TypeScript strictモードを全面適用し、`any`の使用を原則禁止する |
| テスト | ユニットテスト（Vitest）・重要導線の結合テストを整備し、CIで自動実行する |
| エラーハンドリング | ユーザー起因のエラー（入力不備）とシステムエラーを区別し、ユーザーには分かりやすい日本語メッセージを表示する |
| ログ | 構造化ログ（JSON）をサーバー側に出力し、障害調査を可能にする |
| バックアップ | PostgreSQLの定期バックアップ（日次）とリストア手順の整備 |
| 監査ログ | 個人情報の作成・更新・削除・閲覧・ダウンロードを誰がいつ行ったか記録する |
| パフォーマンス | 顧客一覧検索は代表的なデータ量（数千件）で体感1秒以内を目標とする |
| アクセシビリティ | フォームのラベル付け、キーボード操作、コントラスト比等、基本的なa11yを満たす |
| 対応デバイス | PCブラウザを主とし、スマートフォンでも主要機能（閲覧・簡易入力）が利用できる |
| 表示モード | ライト/ダークモードに対応する |

## 3. スコープ外（本フェーズでは扱わない）

顧客ポータル、電子署名、請求書発行、Stripe決済、LINE通知、メール自動送信（定期届出以外の通知）、
チャット、外部API連携。
